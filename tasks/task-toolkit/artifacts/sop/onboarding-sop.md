# 接单 SOP — 从沟通到交付

> **目标：** 标准化接单流程，减少遗漏，提升交付质量
> **适用：** 所有自由职业 / 接单项目

---

## 流程总览

```
沟通 → 评估 → 方案 → 开发 → 交付 → 售后
 ①      ②      ③      ④      ⑤      ⑥
```

---

## ① 沟通阶段

### 目标
快速理解需求，判断是否接单。

### 动作清单

| 动作 | 要点 |
|------|------|
| 记录需求 | 用录音/笔记工具记录客户原始描述 |
| 引导提问 | 5W1H — Who / What / When / Where / Why / How |
| 明确边界 | 什么是"要做的"，什么是"不做的" |
| 确认预算范围 | 在报价前了解客户预算区间 |
| 判断接单 | 参考下方**接单评估表** |

### 接单评估表

| 维度 | 绿灯 ✅ | 黄灯 ⚠️ | 红灯 ❌ |
|------|---------|----------|---------|
| 需求清晰度 | 需求明确 | 部分模糊 | 完全说不清 |
| 技术栈匹配 | 完全匹配 | 部分可学 | 全新领域 |
| 预算匹配 | 高于底线 | 接近底线 | 远低于底线 |
| 排期可行性 | 充裕 | 略紧 | 不可能 |
| 客户配合度 | 主动配合 | 一般 | 难以沟通 |

> **决策：** 2 个以上红灯 → 不接 / 加价 / 转介

---

## ② 评估阶段

### 目标
产出精确的工时和报价估算。

### 动作清单

- [ ] 拆解功能点（按页面/接口/逻辑维度）
- [ ] 标注技术难点
- [ ] 估算工时（乐观/悲观/最可能）
- [ ] 预留 buffer（建议 20-30%）
- [ ] 确定第三方成本（API / 服务器 / 证书等）
- [ ] 输出报价（参考 `proposal-templates/quote-template.md`）

---

## ③ 方案阶段

### 目标
客户确认方案和报价，锁定范围。

### 动作清单

- [ ] 撰写方案书（参考 `proposal-templates/proposal-template.md`）
- [ ] 明确交付物清单
- [ ] 确定排期里程碑
- [ ] 客户确认签字
- [ ] 收取首付款（建议 30-50%）

---

## ④ 开发阶段

### 动作清单

- [ ] 搭建项目脚手架
- [ ] 建立 Git 仓库，初始化提交
- [ ] 按功能点分阶段开发
- [ ] 每完成一个功能点做一次自测
- [ ] 阶段性向客户同步进展
- [ ] 需求变更走变更流程（见下方）

### 需求变更流程

```
客户提出变更
    ↓
评估影响范围 + 工时 + 费用
    ↓
书面告知客户
    ↓
客户确认 → 继续 / 客户取消 → 不做
```

---

## ⑤ 交付阶段

### 动作清单

- [ ] 完成自测（见 `delivery-checklist.md`）
- [ ] 部署到正式环境
- [ ] 提供使用文档 / 演示
- [ ] 客户验收测试
- [ ] 客户确认验收
- [ ] 收取尾款

---

## ⑥ 售后阶段

### 动作清单

| 时间 | 动作 |
|------|------|
| 交付后第 1 天 | 确认客户使用正常 |
| 交付后第 3 天 | 收集反馈 |
| 交付后第 7 天 | 解决初期问题 |
| 免费维护期内 | 按月回访 |
| 免费维护期结束 | 提出续费/升级方案 |

### 售后边界

- 免费维护范围：Bug 修复、环境故障排查
- 额外收费范围：新增功能、内容变更、第三方接口变更适配

## 附录：沟通模板话术

### 需求确认话术

> "收到，我理解您的需求是 [复述需求]。有以下几点需要确认：
> 1. [问题一]
> 2. [问题二]
> 3. [问题三]
> 确认后我会出一份方案和报价给您。"

### 报价跟进话术

> "方案和报价已发您，包含 [核心内��]。
> 如果预算有偏差，也可以调整范围，您看什么时候方便沟通？"
